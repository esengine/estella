# Estella Architecture

This document describes the architecture and design decisions of the Estella 2D
game engine. It is **descriptive of the current code**. For the
*target* architecture, the root-cause rationale behind the current design, and
the forward roadmap, see the architecture-of-record companions:

- [`REARCHITECTURE.md`](./REARCHITECTURE.md) — root-cause re-architecture (RC1–RC6, F1–F4).
- `REARCH_2D_PARITY.md` — 2D feature-parity / modernization roadmap (local, gitignored).

## Overview

Estella is a 2D game engine with a **TypeScript SDK** driven by a
**C++/WebAssembly** backend. It targets web browsers and WeChat MiniGames and
ships with an Electron-based visual editor.

The architecture rests on two non-negotiable principles (see `REARCHITECTURE.md`):

1. **Single source of truth** — a given fact (memory layout, type id, enum value)
   is defined once. Cross-boundary layout is generated by EHT and enforced with
   compile-time `static_assert` + a runtime ABI-hash handshake; drift is a build
   or startup failure, never silent corruption.
2. **One channel per abstraction** — `GfxDevice` for the GPU, a single
   `SparseSet` for component storage, `WasmBridge` for the WASM boundary,
   per-App resources for subsystem state. Bypassing an abstraction should fail to
   compile or fail a CI guard.

```
┌────────────────────────────────────────────────────────────────────┐
│  TypeScript SDK  (sdk/src)                                           │
│  App · World · Schedule · Plugins · Query/Commands/Res/Events        │
│  rendering submit · physics · spine · tilemap · ui · audio · scene   │
└───────────────┬──────────────────────────────────────────────────────┘
                │  WasmBridge  (single boundary; EHT-generated layout/ABI)
┌───────────────▼──────────────────────────────────────────────────────┐
│  C++ / WebAssembly  (src/esengine)                                    │
│  Registry (SparseSet) · RenderFrame → GfxDevice · ResourceManager    │
│  particle · tilemap · text · animation · platform                    │
└────────────────────────────────────────────────────────────────────┘
```

## Core Modules (`src/esengine/`)

### Core (`core/`)

- **Types.hpp** — base types (`u32`, `f32`, `Entity`), and the **domain-scoped
  TypeId** system. Instead of one global `getTypeId<T>()` counter shared across
  unrelated concerns, each domain (`ComponentDomain`, `EventDomain`,
  `ServiceDomain`, `ResourceDomain`) has its own dense, 0-based sequence
  (`Types.hpp:189-219`). Component ids stay dense, so they index the `pools_`
  vector and the per-entity component bitmask without waste or overflow (RC4).
- **Log.hpp** — leveled logging. Note: `ES_ASSERT` compiles out in release;
  boundary validation that must always run does **not** rely on it (RC3).
- **PackedId.hpp** — entity index/generation packing.
- **Engine / Application** — engine singleton, version, lifecycle hooks.

### ECS (`ecs/`)

Entity-Component-System inspired by EnTT, with a **single** storage core (RC2).

```
┌──────────────────────────────────────────────────────────────┐
│                         Registry                              │
│  Entity management: create() / destroy() / valid()           │
│  Component pools: one SparseSet<T> per component type         │
│  Single destroy path: mask-driven, no parallel storages       │
└──────────────────────────────────────────────────────────────┘
```

#### SparseSet — the one storage core

There is exactly one component-storage implementation. The earlier parallel
`DynamicComponentPool` / `SchemaComponentPool` were dead/near-dead code and were
deleted (RC2); JS zero-copy access already read the typed pool's pointer
directly.

```cpp
class SparseSetBase {
    virtual u32 version() const = 0;   // pointer-staleness generation
};

template<typename T>
class SparseSet : public SparseSetBase {
    std::vector<Entity> sparse_;     // Entity → dense index (paged)
    std::vector<Entity> dense_;      // dense index → Entity
    std::vector<T>      components_;  // contiguous component data
    u32 version_;                    // bumped on realloc/relocate
    u32 version() const override { return version_; }  // SparseSet.hpp:370
};
```

- O(1) add / remove / lookup; cache-friendly dense iteration; swap-and-pop delete.
- `version()` (`SparseSet.hpp:97,370`) is the **single source** for pointer
  invalidation: it increments whenever the component buffer reallocates
  (emplace growth) or relocates (remove swap-pop, sort/rebuild). It is the guard
  any caller that *caches* a cross-boundary pointer must re-check.

#### Views / Queries

`registry.view<A, B>()` iterates entities holding all requested components,
filtered via the per-entity component mask. The SDK exposes this as
`Query(...)` with `Mut`, change-detection, and resource/event parameters.

### Renderer (`renderer/`)

WebGL2-based, compiled to WASM. There is **one rendering path** (RC5): the legacy
`Renderer` / `BatchRenderer2D` classes were removed. Everything goes through
`RenderFrame`, and **all** GPU access goes through `GfxDevice`.

```
┌──────────────────────────────────────────────────────────────┐
│  RenderFrame  (per-frame orchestration, frustum cull, stats)  │
│      │ collects RenderItems via typed RenderTypePlugins:      │
│      │   BatchPlugin · SpritePlugin · ParticlePlugin ·        │
│      │   ShapePlugin · TextPlugin · TilemapRenderPlugin ·     │
│      │   UIElementPlugin                                      │
│      ▼                                                        │
│  DrawList  (sort + coalesce; multi-texture batch; u32 index)  │
│      ▼                                                        │
│  GfxDevice  (the only GPU entry point)                        │
│    └── GLDevice : public GfxDevice   (the one concrete impl)  │
│  Shader · Texture · Buffer/VAO · Framebuffer · CustomGeometry │
│  all take GfxDevice&; no raw gl* outside GLDevice (CI-guarded)│
└──────────────────────────────────────────────────────────────┘
```

Key facts:
- **`GfxDevice`** (`GfxDevice.hpp:48`) is an abstract device that owns its
  pipeline state; `GLDevice` (`GLDevice.hpp:33`) is the single concrete backend.
  A CI guard rejects raw `gl*` calls outside `GLDevice.cpp` so the abstraction
  stays the only channel. A `MockGfxDevice` harness exercises it headlessly.
- **`DrawList`** sorts and coalesces draw commands, merging up to 8 textures per
  call; indices are **u32** (no >65535 vertex wraparound).
- **`RenderFrame`** owns transient buffers (`TransientBufferPool`), frustum
  culling, render-to-texture (`RenderTarget`), masking (`RenderFrameMask`:
  scissor + stencil), optional post-processing (`PostProcessPipeline`,
  ping-pong FBOs), material UBOs (`MaterialStore`), and 2D lighting
  (`LightStore` / `LightConstants`, up to 16 dynamic lights with a Lit2D shader
  domain). `ImmediateDraw` provides debug/shape primitives on the same buffers.

### Resource pipeline (`resource/`)

- **ResourceManager / ResourcePool** — reference-counted GPU resources with a
  three-state lifecycle (Held → Evictable → Evicted). An intrusive LRU list plus
  a byte budget (`setTextureBudget`) evicts the oldest unreferenced entries when
  resident bytes exceed the budget. (The budget/stats are not yet surfaced to the
  SDK — see `REARCH_2D_PARITY.md` T1.)

### Other C++ subsystems

`particle/` (CPU simulation, GPU-instanced rendering), `tilemap/`, `text/`
(SDF/glyph), `animation/`, `data/`, `math/` (glm-backed), `platform/`.

## SDK / WASM boundary (`sdk/src/`)

The SDK is the authored surface: `defineComponent`, `defineSystem`, `Query`,
`Commands`, `Res`, events, `Schedule`, and `Plugin`-based composition (`App`).
Subsystems (asset, physics, spine, tilemap, ui, audio, scene, prefab, camera,
particle, postprocess, timeline, net) are plugins.

- **One WASM boundary**: `WasmBridge<M>` (`sdk/src/WasmBridge.ts:57`) is the
  single base class — one loader, one ready promise, one missing-binding policy,
  one error path, with an abort guard. `CoreApiBridge`, `PhysicsBridge`, and
  `SpineModuleBridge` all extend it (F2). The five ad-hoc bridge styles described
  in `REARCHITECTURE.md` have been consolidated here.
- **Generated layout, no hand mirrors**: EHT (`tools/eht`) generates embind,
  pointer accessors, the TS component layout, cross-boundary enums, and the ABI
  layout hash. C++ carries `static_assert(offsetof==N)` for every component
  field; the bridge compares the ABI hash on `connect()` and refuses to start on
  mismatch (RC1).
- **Lifecycle/error model**: `-fno-exceptions` in C++ + explicit status at the
  boundary + a fatal `onAbort` death flag (`moduleHealth.ts`); transient
  `_malloc` is RAII-scoped via `withScratch()` (RC3).
- **Per-App state (F3)**: Camera (`CameraView`), Timeline, PostProcess, and
  SpriteAnimator are per-App/per-World resources, so isolated realms (e.g. the
  editor's play realm) never alias process-global state.

## Platform Integration

```
JavaScript (SDK)                 C++ (WASM)
   es_init(w, h) ───────────────►
   requestAnimationFrame ◄────────
   update(dt) ──────────────────►   World.systems → Registry
   render() ────────────────────►   RenderFrame → GfxDevice
```

- **Web/Emscripten** — WebGL2 context, HTML5 events, canvas management.
- **WeChat MiniGame** — a platform adapter (`sdk/src/platform/wechat`) wraps
  `wx.*` APIs (`WXWebAssembly`, sync file I/O, storage, a dedicated audio
  backend) behind the same `PlatformAdapter` interface used by web.

## Scene Management (SDK)

`SceneManager` is a resource over the single ECS World. Scenes share one World;
entities are tagged with `SceneOwner` to track ownership. The render path filters
cameras and draw callbacks by active scene; scene systems are gated to run only
when their scene is `running`. `sceneManagerPlugin` runs the transition update
each frame. Prefabs expand to tagged entities on load and collapse on save, using
the same expand/collapse core in the SDK as the editor.

## Performance Considerations

1. **Component iteration** — views touch only matching entities; dense arrays.
2. **Memory locality** — components stored contiguously in `SparseSet`.
3. **Batching** — `DrawList` sorts + coalesces; multi-texture per draw; u32 index.
4. **Minimal allocations** — entity recycling, transient buffer pools, RAII scratch.
5. **Eviction** — `ResourcePool` LRU + byte budget bounds VRAM.

## Extension Points

1. **Custom components** — any registered struct (`ES_COMPONENT`/`ES_PROPERTY`).
2. **Custom systems / plugins** — `defineSystem` + `App.addPlugin`.
3. **Custom shaders / materials** — GLSL with reflected `#pragma param` UBO layout.
4. **Custom render types** — implement a `RenderTypePlugin`.
5. **Platform adaptation** — implement `PlatformAdapter`.
